# 牌桌 UI 系统 (Table UI System)

> **Status**: In Design
> **Author**: user + agents
> **Last Updated**: 2026-04-24
> **Implements Pillar**: Presentation — renders all game systems to the player

## Overview

牌桌 UI 系统是《决胜21点》的视觉与交互表现层——它将游戏所有底层系统的数据状态渲染为玩家可感知、可操作的牌桌界面：玩家的手牌（花色、点数、印记、卡质的视觉编码）、AI 对手的手牌（明牌可见、暗牌隐藏）、双方的 HP 血条和防御条、筹码余额和变动、回合阶段指示器（当前处于 8 阶段中的哪一步）、以及所有玩家操作入口（要牌/停牌/双倍下注/分牌/保险/排序确认按钮）。在数据层面，牌桌 UI 是纯消费方——它从卡牌数据模型读取卡牌属性用于渲染，从战斗状态系统读取 HP/防御值用于血条，从结算引擎消费结算动画事件，从回合管理消费阶段切换事件，从筹码经济读取余额用于显示——UI 本身不持有也不修改任何游戏状态。在玩家体验层面，牌桌 UI 就是游戏本身——玩家永远在看着它、点击它、通过它与游戏世界交互。卡牌的翻转动画、伤害数字的弹出、血条的下降、阶段切换的过渡效果，这些视觉反馈将底层系统的逻辑计算转化为玩家可感知的情感冲击。

## Player Fantasy

**牌桌的主宰 — 一眼即知，不疑即行动**

你坐在庄家对面。牌桌是你的领地——每一张牌、每一滴血、每一个筹码、每一个阶段指示器都在你的视野中同时存在，清晰可读，无需思考。UI 的力量不在于它做了什么，而在于它让你做什么：不问即可知，不疑即行动。当对手的明牌翻转，你已经看到了自己的 HP、防御、筹码、手牌点数、可用操作按钮——所有信息在一帧之内同步到位。这是赌场老手的感觉：不需要看第二眼，第一眼就是全部。

核心时刻：要了一张牌，方片 K 翻转。在卡牌落桌的同一秒，你已经扫到 HP 条没动（对手还没出牌），防御值还够，手牌点数跳到 19，"停牌"按钮亮着，"双倍下注"灰掉——一切同步，无需等待，无需猜测。你不是在操作界面，你是在读一场棋局。

这个幻想承接了回合管理的"赌局的心跳"——心跳的弧线由阶段切换驱动，但弧线的每个节点能让你保持冷静判断的前提是：你在任何时刻都拥有完整的信息图景。UI 让赌桌上的信息优势变成真正的战术优势。从排序阶段的结算位预览（"你的黑桃在位1会挡住对手位1的什么"），到结算动画中的伤害流向（HP 条下降、防御值消耗），到阶段指示器告诉你"下一步会发生什么"——UI 的每一个像素都在服务同一个目标：让你在信息充分的条件下做出决策。

## Detailed Design

### Core Rules

**1. 系统性质**

牌桌 UI 系统是一个纯表现层——它从游戏系统读取状态并渲染为视觉界面，接受玩家输入并转发为请求信号。UI 不持有也不修改任何游戏状态。所有状态变化由底层系统驱动，UI 通过信号观察并反映。

**2. 屏幕布局 (1920×1080, 16:9)**

```
┌──────────────────────────────────────────────────┐
│  [对手名/序号]  ████████ HP ████████  防御: 0     │  ← 对手信息栏
│                                                  │
│     [暗牌] [方片Q] [暗牌] [暗牌]                  │  ← 对手手牌区
│                                                  │
│  ┌──────────────────────────────────────────┐    │
│  │  阶段: HIT_STAND  │ 对手: 3/8  │ 回合: 4  │    │  ← 中央信息栏
│  │      筹码: 450  │  先手: 玩家            │    │
│  └──────────────────────────────────────────┘    │
│                                                  │
│     [♠9] [♥J] [♦K] [♦7] [♣3]                   │  ← 玩家手牌区
│       1     2     3     4     5                  │     (排序阶段显示位号)
│                                                  │
│  ████████ HP ████████  防御: 8  点数: 19         │  ← 玩家信息栏
│                                                  │
│  [ 要牌 ] [ 停牌 ] [ 双倍 ] [ 分牌 ] [ 保险 ]    │  ← 操作按钮栏
└──────────────────────────────────────────────────┘
```

**五大区域**:

| 区域 | 位置 | 内容 | 永久可见 |
|------|------|------|---------|
| 对手信息栏 | 顶部 | 对手名称/序号、HP 血条、防御值 | 是 |
| 对手手牌区 | 上部 | AI 手牌（明牌可见、暗牌背面） | 是（牌数=手牌数） |
| 中央信息栏 | 中部 | 阶段指示器、对手序号/8、回合计数、筹码余额、先手方标识 | 是 |
| 玩家手牌区 | 下部 | 玩家手牌（全部可见），排序阶段显示结算位号 | 是 |
| 玩家信息栏+操作栏 | 底部 | HP 血条、防御值、手牌点数；阶段对应操作按钮 | 是（按钮按阶段启用/禁用） |

**3. 卡牌渲染 (CardView)**

每张卡牌是一个 `CardView.tscn` 实例 (~120×168px)。使用纹理图集（texture atlas）共享花色符号、印记图标、卡质图标的纹理资源，将 GPU draw calls 控制在 100 以内。

| 视觉元素 | 来源数据 | 显示规则 |
|---------|---------|---------|
| 花色符号 | `CardInstance.prototype.suit` | 始终显示（红桃/方片/黑桃/草花） |
| 点数文字 | `CardInstance.prototype.rank` | 始终显示（A/2-10/J/Q/K） |
| 印记图标 | `CardInstance.stamp` | stamp ≠ null 时显示（短剑/护盾/心/钱币/重锤/跑鞋/乌龟） |
| 卡质边框 | `CardInstance.quality` | quality ≠ null 时显示（铜/银/金/钻/宝石色） |
| 卡质图标 | `CardInstance.quality` + `quality_level` | quality ≠ null 时显示品质+等级 |
| 结算位号 | 排序阶段由排序系统分配 | 仅 SORT 阶段显示（1, 2, 3...） |
| 卡牌背面 | 无（纯视觉状态） | AI 非明牌时覆盖显示 |

**4. 手牌布局算法**

手牌区使用动态间距算法，根据手牌数量自动调整卡牌间距：
- 1-3 张：无重叠，均匀分布
- 4-7 张：适度重叠（每张偏移 ~80px）
- 8-11 张：紧密重叠（每张偏移 ~55px），边缘卡牌部分可见

分牌时，玩家手牌区分裂为左右两个子区域，各显示一手牌。

**5. 阶段对应 UI 行为**

| 阶段 | 可见操作按钮 | 卡牌交互 | 中央信息栏 |
|------|------------|---------|-----------|
| DEAL | 无 | 不可交互（发牌动画） | "发牌中..." |
| SIDE_POOL | 无（MVP 跳过） | 不可交互 | 跳过 |
| INSURANCE | [购买保险(30筹码)] [购买保险(6HP)] [跳过] | 不可交互 | "是否购买保险？" + AI 明牌高亮 |
| SPLIT | [确认分牌] [跳过] | 不可交互 | "是否分牌？" |
| HIT_STAND | [要牌] [停牌] [双倍下注] | 不可交互 | 手牌点数高亮显示 |
| SORT | [确认排序] | 拖拽排序 + 30秒倒计时 | "排列你的卡牌" + 倒计时 |
| RESOLUTION | 无 | 不可交互（结算动画） | 当前结算步骤描述 |
| DEATH_CHECK | 无 | 不可交互 | 回合结果 |

**6. 操作按钮规则**

- 按钮按阶段启用/禁用，不可用按钮灰显（`disabled = true`）。
- 双倍下注：仅在手牌恰好 2 张且未分牌时启用。
- 分牌：仅在起始两张牌 rank 相同时启用。
- 保险：仅在 AI 明牌为 A 时启用。筹码不足 30 且 HP 不足 6 时，对应支付方式的保险按钮禁用。
- 游戏手柄：阶段切换时自动聚焦第一个可用按钮（`grab_focus()`）。

**7. 分牌 UI 布局**

分牌触发时，玩家手牌区分裂为两个并排子区域：

```
  ┌─── 手牌 A ───┐  ┌─── 手牌 B ───┐
  │ [♠9] [♥J]    │  │ [♦7] [♣3]    │
  │   1     2    │  │   1     2    │
  └──────────────┘  └──────────────┘
     当前结算 ▲         等待结算
```

- 手牌 A 先结算（高亮边框），手牌 B 暗淡。
- 手牌 A 结算完成后，B 高亮，A 暗淡。
- AI 手牌区同理分牌（如 AI 也分牌）。
- 操作栏在每手牌的 HIT_STAND 阶段分别激活。

### States and Transitions

```
TABLE_IDLE ──(DEAL开始)──→ DEALING
DEALING ──(发牌动画完成)──→ INSURANCE_CHECK
INSURANCE_CHECK ──(AI明牌≠A或玩家跳过)──→ SPLIT_CHECK
INSURANCE_CHECK ──(玩家购买)──→ INSURANCE_RESULT → SPLIT_CHECK
SPLIT_CHECK ──(不可分牌或玩家跳过)──→ HIT_STAND
SPLIT_CHECK ──(确认分牌)──→ SPLIT_ACTIVE → HIT_STAND_A
HIT_STAND_A ──(停牌)──→ HIT_STAND_B (仅分牌时)
HIT_STAND_B ──(停牌)──→ SORT
HIT_STAND ──(双方停牌,无分牌)──→ SORT
SORT ──(确认或倒计时结束)──→ RESOLUTION
RESOLUTION ──(结算动画完成)──→ DEATH_CHECK
DEATH_CHECK ──(CONTINUE)──→ TABLE_IDLE
DEATH_CHECK ──(PLAYER_WIN)──→ VICTORY_TRANSITION → 商店UI
DEATH_CHECK ──(PLAYER_LOSE)──→ GAME_OVER_SCREEN
```

UI 状态标识：
- **活跃区域高亮**：当前可交互的区域（手牌/按钮）显示高亮边框。
- **非活跃区域暗淡**：不可交互区域降低透明度（alpha ~0.5）。

| 战斗状态系统 (#7) | `hp`, `max_hp`, `defense` → 血条/防御值渲染 | 无 |
| 回合管理 (#13) | `phase_changed` → 阶段切换、按钮启用/禁用；`first_player` → 发牌先手方标识；`settlement_first_player` → 结算先手方标识；`opponent_number` → 对手序号；`round_counter` → 回合计数 | `player_hit_requested`, `player_stand_requested`, `player_double_down_requested`, `player_split_requested`, `player_insurance_requested`, `player_sort_confirmed` |

| 点数计算引擎 (#2) | 手牌点数结果 → 点数显示 | 无（由回合管理中转） |
| 牌型检测 (#3) | 牌型名称和倍率 → 排序阶段预览 | 无 |
| 卡牌排序 (#6a) | 排序确认信号 → 结算位号更新 | `player_sort_confirmed(manual_order)` |
| 特殊玩法 (#8) | 保险/分牌/双倍条件 → 按钮可用性 | 按钮点击 → 通过回合管理转发 |
| AI 对手 (#12) | AI 行动（通过回合管理）→ AI 区域动画 | 无 |
| 对局进度 (#14) | `opponent_number`, `total_opponents` → 进度显示 | 无 |

**信号架构原则**:
- UI 控制器连接游戏逻辑节点的信号（`phase_changed`, `hp_changed`, `cards_dealt` 等），不使用全局事件总线。
- 玩家操作通过 UI 控制器的请求信号转发（`player_hit_requested` 等），游戏逻辑节点连接这些信号。
- UI 永远不直接调用游戏逻辑方法——单向依赖：UI → 数据类型（读取），UI → 请求信号（发出）。

## Formulas

### 1. 手牌间距计算 (card_spacing)

根据手牌数量动态调整卡牌之间的水平偏移：

```
card_spacing(hand_count, container_width, card_width):
  IF hand_count <= 3:
    spacing = (container_width - hand_count × card_width) / (hand_count + 1)
  ELIF hand_count <= 7:
    spacing = 80  # 固定重叠偏移
  ELSE:
    spacing = (container_width - card_width) / (hand_count - 1)
```

| 变量 | 类型 | 范围 | 说明 |
|------|------|------|------|
| `hand_count` | int | 1-11 | 手牌数量 |
| `container_width` | int | ~1600px | 手牌区可用宽度 |
| `card_width` | int | 120px | 卡牌宽度 |

**输出范围**: spacing ∈ [55, 500]px。最大间距 = 单张牌居中。

### 2. 血条填充比 (hp_bar_ratio)

```
hp_bar_ratio = hp / max_hp
bar_fill_width = hp_bar_ratio × bar_total_width
```

| 变量 | 类型 | 范围 | 说明 |
|------|------|------|------|
| `hp` | int | 0-300 | 当前 HP |
| `max_hp` | int | 80-300 | HP 上限 |

**输出范围**: [0.0, 1.0]。玩家 max_hp=100，AI max_hp 按 `ai_hp_scaling` 表。

### 3. 排序倒计时 (sort_countdown)

```
time_remaining = sort_timer_seconds - elapsed_since_sort_start
display_seconds = ceil(time_remaining)
```

**输出范围**: [0, 30]秒。倒计时归零时自动触发 `player_sort_confirmed`。

### 4. 筹码变动动画值 (chip_animation_value)

```
display_value = animate(current_display, target_balance, duration=0.5s, easing=ease_out)
```

筹码余额变化时，显示数字在 0.5 秒内平滑滚动到新值。使用 Tween 驱动。

## Edge Cases

- **手牌达上限（11 张）时的显示**: 动态间距算法将卡牌紧密重叠至 ~55px 偏移。最左侧和最右侧卡牌完全可见，中间卡牌仅露出花色/点数的左上角区域。悬停（鼠标）或聚焦（手柄）时该卡牌上浮 20px，完整展示。所有 11 张牌的结算位号始终可见（位号标签位于卡牌底部独立区域，不被重叠遮挡）。

- **AI 手牌全部可见（结算阶段）**: 结算阶段 AI 所有暗牌翻转。翻转使用 Tween 动画（缩放 X 轴至 0 → 切换纹理 → 缩放回 1），每张牌间隔 200ms 依次翻转。11 张 AI 牌翻转耗时 ~2.2 秒。

- **同时显示分牌双手 + AI 分牌双手**: 最坏情况 4 个手牌区（玩家 A/B + AI A/B）。玩家手牌区左右分裂，AI 手牌区同理。每个子区域宽度减半（~800px），卡牌间距重新计算。11 张牌在 800px 内偏移 ~55px 仍然可行。结算时仅活跃手牌区高亮，其余暗淡。

- **结算动画期间玩家试图操作**: RESOLUTION 阶段所有按钮禁用，所有卡牌不可交互。结算动画队列完成后，UI 切换到 DEATH_CHECK 状态。玩家的任何输入（点击/按键）在动画期间被忽略。

- **排序阶段倒计时归零**: 自动以当前排列顺序触发 `player_sort_confirmed`。倒计时最后 5 秒文字变红并放大闪烁。不弹出确认对话框。

- **筹码超过 cap（999）时的显示**: 筹码余额显示截断为 999。溢出部分由筹码经济系统在逻辑层截断，UI 只需正确显示系统提供的余额值。如果动画中间值超过 999（滚动动画），在动画结束时 snap 到实际值。

- **HP 降至 0 时的视觉反馈**: 血条清空时短暂闪烁红色（0.3 秒），然后灰色。如果后续红桃回复将 HP 恢复到 0 以上，血条重新填充并变绿。

- **保险购买后 AI 揭牌**: 保险触发时，AI 的暗牌临时翻转显示（1.5 秒），然后翻回暗牌状态。如果保险有效（对手为杰克/黑杰克），暗牌保持翻开直到结算阶段。

- **宝石质摧毁动画**: 卡牌结算中摧毁检查失败时，卡质边框碎裂动画（缩小+淡出，0.5 秒），卡质图标消失。卡牌本身保留，仅移除品质视觉元素。

- **窗口缩放**: UI 使用 Godot 的 anchor + container 系统响应窗口大小变化。最小支持分辨率 1280×720。低于此分辨率时卡牌缩小至 ~90px 宽。

## Visual/Audio Requirements

**视觉要求:**
- 卡牌花色使用标准色彩编码：红桃/方片 = 暖红色，黑桃/草花 = 冷黑色。
- 印记图标使用差异化形状+颜色：短剑(橙色)、护盾(蓝色)、心(粉色)、钱币(金色)、重锤(灰色)、跑鞋(绿色)、乌龟(棕色)。
- 卡质边框颜色分级：铜(铜色) → 银(银色) → 金(金色) → 钻(钻石蓝白) → 宝石质(对应花色加强色)。
- HP 血条：满血绿色，50%以下黄色，25%以下红色闪烁。
- 防御值：蓝色数值标签，与 HP 血条分离显示。
- 结算动画颜色通道：玩家卡牌效果绿色/蓝色，AI 卡牌效果红色/橙色。
- 伤害弹出数字：红色上浮消失。回复弹出：绿色。防御弹出：蓝色。筹码弹出：金色。

**音频要求:**
- 发牌音效：每张牌一张清脆的"啪"声（卡牌落桌）。
- 翻牌音效：短促的翻转声（AI 揭牌）。
- 按钮点击音效：柔和的确认音。
- 伤害音效：受击低沉音（方片伤害）。
- 回复音效：轻柔上升音（红桃回复）。
- 防御音效：金属碰撞音（黑桃防御）。
- 筹码音效：硬币叮当声（草花筹码）。
- 爆牌音效：沉重的冲击声（爆牌自伤）。
- 宝石摧毁音效：碎裂声（品质碎裂动画）。
- 胜利音效：短促的胜利号角。
- 失败音效：低沉的落败鼓声。
- 音量和音效具体规格由音频总监在实现阶段定义。

## UI Requirements

**无障碍 (Accessibility):**
- 所有颜色信息同时以图标/文字形式传达（不依赖颜色区分）。
- 字体最小 14px，关键数字（HP/点数）最小 18px。
- 支持键盘完全操作：Tab 切换按钮焦点，Enter 确认，方向键排序。
- 游戏手柄完全操作：A 确认，B 取消，肩键切换手牌区。
- 利用 Godot 4.5+ AccessKit 支持屏幕阅读器。

**本地化 (Localization):**
- 所有可见文字使用 `tr()` 包裹。
- 按钮文字和阶段描述使用自动换行（`AUTOWRAP_WORD_SMART`）。
- 花色/印记名称支持翻译（中文、英文）。

**输入响应:**
- 鼠标点击响应延迟 <50ms。
- 按钮悬停状态即时显示。
- 排序拖拽实时跟随鼠标（无延迟）。
- 阶段切换动画不超过 300ms（不阻塞游戏逻辑）。

## Acceptance Criteria

**AC-01: 屏幕布局完整性**
GIVEN 游戏初始化完成，进入第一个对手的第一个回合
WHEN 牌桌 UI 渲染完成
THEN 五大区域全部可见：对手信息栏（顶部）、对手手牌区（上部）、中央信息栏（中部）、玩家手牌区（下部）、玩家信息栏+操作栏（底部）。每个区域不为空。

**AC-02: 卡牌视觉渲染**
GIVEN 玩家手牌包含 [♠A(stamp=SHIELD, quality=OBSIDIAN/II), ♥K(null, null), ♦7(stamp=COIN, quality=null)]
WHEN CardView 实例渲染这三张牌
THEN ♠A 显示黑桃符号+"A"+护盾图标+黑曜石边框+II 级标记；♥K 显示红桃符号+"K"，无印记/卡质图标；♦7 显示方片符号+"7"+钱币图标，无卡质边框。

**AC-03: AI 明牌/暗牌显示**
GIVEN AI 手牌 4 张，第一张为明牌（♦Q），其余 3 张暗牌
WHEN 渲染 AI 手牌区
THEN 第一张牌显示花色+点数（♦Q），其余 3 张显示卡牌背面。4 张牌按手牌数量排列。

**AC-04: HP 血条渲染**
GIVEN 玩家 HP=75, max_hp=100; AI HP=200, max_hp=300
WHEN 渲染双方血条
THEN 玩家血条填充 75%（绿色），AI 血条填充 ~66.7%（绿色）。血条旁显示数字 "75/100" 和 "200/300"。

**AC-05: 阶段切换按钮启用/禁用**
GIVEN 当前阶段为 HIT_STAND，玩家 2 张牌（未分牌），手牌点数 11
WHEN 渲染操作栏
THEN [要牌] 启用，[停牌] 启用，[双倍下注] 启用（2 张牌且未分牌），[分牌] 禁用（非对子），[保险] 禁用（非 INSURANCE 阶段），[确认排序] 禁用（非 SORT 阶段）。

**AC-06: 保险按钮条件禁用**
GIVEN AI 明牌=A，进入 INSURANCE 阶段；玩家筹码=20, HP=4
WHEN 渲染保险操作
THEN [购买保险(30筹码)] 禁用（筹码不足），[购买保险(6HP)] 禁用（HP不足），[跳过] 启用。

**AC-07: 排序拖拽**
GIVEN SORT 阶段，玩家手牌 5 张
WHEN 玩家将第 3 张牌拖拽到第 1 位置
THEN 卡牌视觉顺序更新为 [原3, 原1, 原2, 原4, 原5]，结算位号更新为 [1, 2, 3, 4, 5]。30 秒倒计时显示。

**AC-08: 排序倒计时自动确认**
GIVEN SORT 阶段，倒计时剩余 0 秒
WHEN 倒计时归零
THEN 自动以当前排列触发 `player_sort_confirmed`，UI 切换到 RESOLUTION 阶段。无确认弹窗。

**AC-09: 手牌间距动态调整**
GIVEN 玩家手牌分别为 2 张、5 张、11 张
WHEN 渲染手牌区
THEN 2 张：均匀分布无重叠；5 张：适度重叠（~80px 偏移）；11 张：紧密重叠（~55px 偏移），所有卡牌至少部分可见。

**AC-10: 分牌 UI 分裂**
GIVEN 分牌触发，玩家手牌分为 [♠9, ♥J] 和 [♦7, ♣3]
WHEN 分牌 UI 激活
THEN 玩家手牌区分裂为左右两个子区域。手牌 A 显示 [♠9, ♥J] 并高亮，手牌 B 显示 [♦7, ♣3] 并暗淡。结算位号分别标记 [1, 2]。

**AC-11: 结算动画交替执行**
GIVEN 结算阶段，玩家 3 张牌，AI 3 张牌，settlement_first_player=PLAYER
WHEN 结算动画执行
THEN 每张牌完整走完结算管道动画（基础值→印记→卡质→倍率→效果→摧毁检查）后，再开始下一张。交替顺序：玩家位1 → AI位1 → 玩家位2 → AI位2 → 玩家位3 → AI位3。每张牌之间间隔 `anim_settle_delay_ms`（默认 500ms）。玩家牌绿色光效，AI 牌红色光效。

**AC-12: 筹码变动动画**
GIVEN 筹码余额从 100 变为 175（草花结算 +75）
WHEN `chips_changed` 信号触发
THEN 筹码数字在 0.5 秒内从 100 平滑滚动到 175。显示 "+75" 金色弹出文字。

**AC-13: HP 降至 0 闪烁**
GIVEN 结算中玩家 HP 从 15 降至 0
WHEN HP 更新渲染
THEN 血条闪烁红色 0.3 秒，然后变为灰色空条。HP 数字显示 "0/100"。

**AC-14: 阶段指示器更新**
GIVEN 回合管理发出 `phase_changed(SORT)`
WHEN UI 接收信号
THEN 中央信息栏阶段文字更新为"排列你的卡牌"，显示 30 秒倒计时。操作栏仅 [确认排序] 启用。

**AC-15: 先手方标识**
GIVEN first_player=AI（玩家为后手）
WHEN 回合开始渲染
THEN 中央信息栏显示先手方为 AI。操作栏在 HIT_STAND 阶段时，AI 先行动（按钮禁用直到 AI 完成），然后玩家按钮启用。

**AC-16: 对手进度显示**
GIVEN opponent_number=4, total_opponents=8
WHEN 中央信息栏渲染
THEN 显示"对手: 4/8"。

**AC-17: 宝石摧毁视觉**
GIVEN 结算中一张红水晶 III 卡牌摧毁检查失败
WHEN 摧毁动画执行
THEN 卡质边框缩小+淡出（0.5 秒），卡质图标消失。卡牌本身保留，花色和点数仍可见。浮动文字显示"品质摧毁"。

**AC-18: 保险揭牌**
GIVEN 玩家购买保险，对手明牌=A，暗牌=K（对手为杰克）
WHEN 保险生效
THEN AI 暗牌翻转显示 K（1.5 秒），然后保持翻开。结算阶段所有 AI 牌正常翻开。

**AC-19: 信号发射验证**
GIVEN 玩家在 HIT_STAND 阶段点击 [要牌] 按钮
WHEN 按钮点击事件触发
THEN `player_hit_requested` 信号从 UI 控制器发出，游戏逻辑节点接收信号并执行要牌逻辑。UI 不直接调用任何游戏逻辑方法。

## Open Questions

1. **商店 UI 是否属于本系统？** — 当前 GDD 范围限于牌桌界面（对战阶段）。商店 UI 是独立的屏幕还是牌桌的叠加层？如果是独立屏幕，需要单独的 GDD（商店系统 #11 的 UI 部分）。负责人：game designer + ux-designer，目标：Alpha 阶段。

2. **对局进度系统 (#14) 提供什么数据？** — 当前 UI 显示"对手: N/8"，但 #14 未设计。是否需要显示更多信息（如历史战绩、当前胜率、下一对手预览）？负责人：game designer，目标：Alpha 阶段。

3. **4.6 双焦点系统的实际测试** — AC 中描述了鼠标悬停（白色内光）和手柄聚焦（黄色边框）的分离视觉。这需要在实际引擎中验证两种输入同时活跃时的视觉效果。负责人：ui-programmer，目标：实现阶段。

4. **结算动画总时长的可接受范围** — 最坏情况：11 张玩家牌 + 11 张 AI 牌，每张 500ms = 11 秒。加上基础值+印记+卡质+倍率的子动画，单张可能需要 2 秒。总时长可能达到 22-44 秒。是否需要快进/跳过功能？负责人：game designer，目标：playtest 阶段。

5. **卡牌背面美术风格** — 需要美术总监定义卡牌背面的视觉设计（纯色/图案/品牌标志）。影响 AI 暗牌区域的视觉密度。负责人：art-director，目标：美术制作阶段。
